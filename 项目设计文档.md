## 一、设计原则与目标

### 1.1 核心目标

构建一套**稳定、隐蔽、可扩展**的跨平台会议中转系统，支持以下场景：

| 方向 | 说明 |
|:---|:---|
| A → Others | 腾讯会议发起，钉钉/飞书/瞩目/华为云/WeLink/Lark 用户参加 |
| Others → A | 上述平台发起，腾讯会议用户参加 |
| N:M 桥接 | 多平台同时接入同一会议（未来扩展） |

**支持平台：** 腾讯会议、钉钉（DingTalk）、飞书（Feishu）、Lark、瞩目（Zhumu）、华为云会议、WeLink

### 1.2 架构设计原则

| 原则 | 说明 |
|:---|:---|
| **最小暴露面** | 控制平面不对公网直接暴露；所有入口经由网关层 |
| **零信任** | 节点间通信全程 mTLS；账号操作全程审计 |
| **无状态工作节点** | Worker 节点不持久化业务数据，便于水平扩展 |
| **故障隔离** | 单个桥接会话崩溃不影响其他会话；单节点故障不影响调度 |
| **流量拟真** | 所有对外网络行为模拟真实用户特征，避免被会议平台识别 |

## 三、账号与多租户系统

### 3.1 租户模型

系统采用**三层租户模型**：

```
Platform（平台）
  └── Organization（组织/企业）
        ├── Team（团队）
        │     └── User（用户）
        └── ApiClient（API 接入方）
```

| 角色 | 权限描述 |
|:---|:---|
| `super_admin` | 平台级：管理所有组织、节点、全局配置 |
| `org_admin` | 组织级：管理本组织用户、配额、平台凭证 |
| `operator` | 发起/停止桥接会话，查看本组织会话 |
| `viewer` | 只读：查看会话状态和历史 |
| `api_client` | 机器账号：通过 API Key 调用，权限范围可定制 |

### 3.2 认证方案

```
┌──────────────────────────────────────────────────────┐
│                   认证流程                             │
│                                                      │
│  用户名/密码 ──→  bcrypt 校验 ──→ 签发 JWT            │
│                                   │                  │
│  SSO/OAuth2   ──→  IdP 重定向  ──→ │                  │
│  (企业微信/飞书/钉钉 OAuth)          │                  │
│                                   ▼                  │
│                         AccessToken (15min)          │
│                         RefreshToken (7d, 存 Redis)  │
│                                   │                  │
│  API Key 调用  ────────────────────┘                  │
│  (HMAC-SHA256 签名验证)                               │
└──────────────────────────────────────────────────────┘
```

**JWT Payload 设计：**

```json
{
  "sub": "user_uuid",
  "org": "org_uuid",
  "role": "operator",
  "scopes": ["bridge:create", "bridge:read"],
  "jti": "unique_token_id",
  "exp": 1712500000
}
```

### 3.3 平台凭证保险库（Credential Vault）

各平台的会议账号密码/Cookie/Token 是系统核心资产，需加密保管。

**加密方案：** AES-256-GCM，密钥通过 HSM（或 HashiCorp Vault）托管，应用层只持有加密密文。

**凭证类型：**

```typescript
interface PlatformCredential {
  id: string;
  orgId: string;
  platform: PlatformType;
  type: 'account' | 'cookie' | 'api_key';

  // 加密存储，解密后使用
  encryptedData: string;   // AES-256-GCM 密文
  iv: string;              // 初始化向量
  keyVersion: number;      // 密钥版本，支持轮换

  displayName: string;     // 如 "飞书-账号A"，明文，用于展示
  lastUsedAt: Date;
  expiresAt?: Date;        // Cookie 过期时间
  isValid: boolean;        // 上次使用是否成功
}
```

**凭证分配策略：**

- 每个组织可注册多个同平台凭证（账号池）
- 调度时按"最近最少使用"（LRU）+ "有效状态"筛选凭证
- 凭证失效自动标记，触发告警通知管理员补充

### 3.4 配额管理

| 配额维度 | 说明 |
|:---|:---|
| `max_concurrent_sessions` | 同时进行的桥接会话上限 |
| `max_sessions_per_day` | 每日桥接次数上限 |
| `max_session_duration_min` | 单次会话最长时长（分钟） |
| `allowed_platforms` | 允许使用的平台组合 |
| `node_region_allowed` | 允许使用的节点区域 |

## 四、内网穿透架构

### 4.1 为什么需要内网穿透

以下两类场景必须解决内网穿透：

1. **Worker 节点在企业内网**：某些企业要求会议连接从内部 IP 发出（安全合规要求），Worker 节点部署在企业内网，需要与云端控制平面通信。
2. **私有化部署**：整套系统部署在客户内网，需要对外连接各会议平台云服务。

### 4.2 穿透架构方案

```
┌─────────────────────────────────────────────────────────────────┐
│  企业内网                                                         │
│                                                                 │
│  ┌─────────────┐    内网隧道（主动出连）    ┌──────────────────┐  │
│  │ Worker Node  │──────────────────────→  │  公网中继服务器   │  │
│  │ (内网机器)   │   WireGuard / frp        │  (穿透服务端)    │  │
│  └─────────────┘                          └────────┬─────────┘  │
│                                                    │            │
└────────────────────────────────────────────────────┼────────────┘
                                                     │
                                           ┌─────────▼──────────┐
                                           │   控制平面           │
                                           │  (Control Plane)   │
                                           └────────────────────┘
```

### 4.3 穿透协议选型

| 方案 | 适用场景 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **WireGuard VPN** | 长期稳定的内网节点 | 内核态，性能极佳；mTLS 天然加密 | 需要 root 权限 |
| **frp（内网穿透）** | 快速接入，无法安装内核模块时 | 无需 root；配置简单 | 性能弱于 WireGuard |
| **SSH 反向隧道** | 临时调试、紧急情况 | 零依赖 | 无自动重连；不稳定 |
| **Cloudflare Tunnel** | 公有云部署，无需固定 IP | 免运维 | 需要信任 Cloudflare |

**推荐方案：** 生产环境用 WireGuard 组建 Overlay 网络；备用方案用 frp。

### 4.4 WireGuard Overlay 网络设计

```
                      WireGuard Hub (控制平面侧)
                      10.100.0.1/24
                           │
             ┌─────────────┼──────────────┐
             │             │              │
        10.100.0.2    10.100.0.3    10.100.0.4
         Worker-A       Worker-B      Worker-C
        (云服务器)     (企业内网A)    (企业内网B)
```

- 每个 Worker 节点启动时向 Hub 注册，获取 WireGuard 配置
- 控制平面通过 Overlay IP 下发任务，与 Worker 通信全程加密
- Worker 节点的公网出口流量走代理池，不经过 WireGuard（避免混淆）

### 4.5 Worker 节点自注册流程

```
Worker 启动
    │
    ▼
生成本地 WireGuard 密钥对
    │
    ▼
POST /api/node/register (携带公钥 + 节点能力描述)
    │ (HTTPS，Bootstrap Token 认证)
    ▼
控制平面分配 Overlay IP，返回 Hub 公钥 + 端点
    │
    ▼
建立 WireGuard 隧道
    │
    ▼
Worker 上线，开始接受任务
```

## 五、隐藏中继基础设施

### 5.1 威胁模型

系统的"隐藏"目标是避免被各会议平台识别出这是一个自动化中转机器人，而非真实用户。识别维度如下：

| 识别维度 | 平台检测手段 | 我们的对策 |
|:---|:---|:---|
| IP 信誉 | 数据中心 IP 直接标记为机器人 | 住宅代理池出口 |
| IP 关联 | 同一 IP 频繁加入不同会议 | 每个会话分配独立出口 IP |
| Browser 指纹 | Canvas/WebGL/字体指纹检测 | 指纹随机化池 |
| 行为模式 | 操作太快/无鼠标移动 | 人性化操作模拟 |
| TLS 指纹 | JA3/JARM 指纹识别自动化工具 | 使用真实 Chrome TLS 栈 |
| 设备指纹 | 摄像头/麦克风型号探测 | 虚拟设备型号随机化 |
| 账号行为 | 账号从未有历史行为 | 账号预热策略 |

### 5.2 多层隐藏架构

```
Worker 内部
    │
    │  请求发出前经过以下层级
    ▼
┌─────────────────────────────────────────┐
│  Layer 1: Browser 指纹随机化              │
│  - 随机 UA、分辨率、时区、语言             │
│  - Canvas 指纹注入噪声                    │
│  - WebRTC IP 泄露屏蔽                     │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Layer 2: 行为拟人化                      │
│  - 随机延迟（正态分布，非均匀分布）          │
│  - 模拟鼠标移动轨迹                        │
│  - 随机打字速度                            │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Layer 3: 出口 IP 隔离                    │
│  - 每个会话独立住宅代理 IP                  │
│  - SOCKS5 代理注入到 Chromium              │
│  - 代理 IP 与账号绑定（避免 IP 切换）        │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Layer 4: 流量混淆（控制平面通信）           │
│  - 控制平面 API 通过 CDN 前置（域前置）      │
│  - 非敏感心跳流量混入正常业务流量             │
└─────────────────────────────────────────┘
```

### 5.3 住宅代理池设计

```typescript
interface ProxyEntry {
  id: string;
  ip: string;
  port: number;
  protocol: 'socks5' | 'http';
  username: string;
  password: string;

  country: string;
  isp: string;            // 运营商（中国电信/联通/移动）
  isResidential: boolean; // 必须为 true

  // 使用状态
  boundAccountId?: string;   // 绑定的平台账号 ID（账号-IP 固定绑定）
  lastCheckAt: Date;
  isHealthy: boolean;
  failCount: number;
}
```

**代理调度策略：**

1. 新会话 → 查找与该平台账号已绑定的代理（账号-IP 固定映射，避免 IP 突变触发风控）
2. 账号无绑定代理 → 从健康代理池中分配一个未绑定的代理，写入绑定关系
3. 代理故障 → 解除绑定，分配新代理，同时对该账号做 1 小时冷却

### 5.5 账号预热策略

新注册的会议账号直接加入会议会触发风控。需执行预热流程：

```
账号注册
    │
    ▼
Day 1-3: 只登录、浏览主页、修改头像（模拟真实用户行为）
    │
    ▼
Day 4-7: 加入几场公开会议，停留 5~10 分钟后退出
    │
    ▼
Day 8+: 账号激活，加入可用凭证池
```

预热行为由 Puppeteer 脚本自动执行，随机时间触发。

### 5.6 控制平面隐藏（域前置）

Worker 节点与控制平面的 API 通信不走裸 HTTPS，而是走 CDN 域前置：

```
Worker → SNI: cdn.example.com (CDN 域名)
        Host Header: api.open-meeting-internal.com (真实控制平面域名)
```

这样即使平台监控 DNS/SNI，也只看到 CDN 域名，无法识别是控制平面通信。

## 六、反追踪体系

### 6.1 反追踪目标

防止以下三类主体追踪到系统的存在：
1. **会议平台方**：识别出中转机器人，封号/封 IP
2. **第三方监控**：追踪流量来源，识别隐藏节点位置
3. **网络层溯源**：通过流量特征还原会话拓扑

### 6.2 会话隔离

每个桥接会话在独立的沙箱中运行，会话间**严格隔离**：

| 隔离维度 | 实现方式 |
|:---|:---|
| 进程隔离 | 每个 Browser 实例独立进程，独立 PID 命名空间 |
| 网络隔离 | 每个会话独立 SOCKS5 代理出口（不同 IP） |
| 文件系统隔离 | 每个 Browser 使用独立 `--user-data-dir` 临时目录 |
| Cookie 隔离 | 会话结束后彻底清理，下次从加密存储重新加载 |
| 内存隔离 | 会话结束后主动释放，防止内存中遗留凭证 |

### 6.3 流量特征混淆

**问题：** FFmpeg + Puppeteer 组合的 WebRTC 流量模式与正常用户有差异。

**对策：**

```
正常用户 WebRTC 特征：
  - 分辨率：动态变化（网络好时提升）
  - 码率：变化范围 200kbps ~ 2Mbps
  - 关键帧间隔：随机

机器人 WebRTC 特征（暴露点）：
  - 分辨率：固定不变
  - 码率：极其稳定（FFmpeg 恒定码率）
  - 没有背景噪声

对策：
  1. FFmpeg 参数加入轻微码率波动（±5%随机）
  2. 视频源加入轻微噪声帧（模拟摄像头噪点）
  3. 音频源注入极低幅度的环境噪声底噪
  4. 随机模拟"网络抖动"（短暂降低质量后恢复）
```

### 6.4 操作行为拟人化

```javascript
// 人性化延迟工具函数（正态分布延迟，非均匀随机）
function humanDelay(mean = 500, stddev = 150) {
  // Box-Muller 变换生成正态分布
  const u1 = Math.random(), u2 = Math.random();
  const z = Math.sqrt(-2 * Math.log(u1)) * Math.cos(2 * Math.PI * u2);
  return Math.max(100, Math.round(mean + z * stddev));
}

// 模拟鼠标移动（贝塞尔曲线轨迹）
async function humanMouseMove(page, targetX, targetY) {
  const steps = 15 + Math.floor(Math.random() * 10);
  // 生成贝塞尔曲线控制点
  // ... 分步移动，每步加入微小抖动
}

// 模拟人工输入（逐字符，随机延迟）
async function humanType(page, selector, text) {
  for (const char of text) {
    await page.type(selector, char, { delay: humanDelay(80, 30) });
  }
}
```

### 6.5 日志与运营数据去标识化

系统内部日志不记录：
- 完整会议号（记录哈希值）
- 平台账号明文（记录脱敏后 ID）
- 用户真实 IP（记录地理区域）

对外 API 响应不返回：
- Worker 节点真实 IP 或主机名
- 使用的代理 IP
- 内部会话 ID（返回不可逆映射后的外部 ID）

## 八、安全架构与信任边界

### 8.1 信任边界图

```
[不信任区域]              [半信任区域]           [完全信任区域]
  
  外部用户 API 调用  ──→   API Gateway      ──→   控制平面内部服务
  
  各会议平台云服务   ──→   Worker（隔离沙箱）──→   mTLS 通信通道
  
  Webhook 回调      ──→   Webhook 验证层   ──→   事件总线
```

### 8.2 密钥与秘密管理

| 类型 | 存储位置 | 访问方式 |
|:---|:---|:---|
| 平台账号密码 | 加密数据库（AES-256-GCM） | Vault Service 解密后临时注入 |
| JWT 签名密钥 | HashiCorp Vault / KMS | 运行时动态获取，不落盘 |
| mTLS 证书 | Vault PKI 自动签发 | 节点启动时获取，24h 轮换 |
| 代理账号密码 | 加密配置文件 | 启动时解密加载到内存 |
| 数据库密码 | 环境变量（非 .env 文件） | 容器启动时注入 |

**原则：任何秘密不以明文形式写入代码、配置文件或日志。**

### 8.3 网络安全策略

```yaml
# 控制平面网络策略（示意）
入站规则:
  - 允许: API Gateway → Control Plane (443/HTTPS)
  - 允许: Worker (Overlay IP) → Control Plane (8443/gRPC+mTLS)
  - 拒绝: 其他所有

Worker 节点网络策略:
  - 出站允许: 经代理池 → 各会议平台域名 (443)
  - 出站允许: Worker → WireGuard Hub (51820/UDP)
  - 出站拒绝: Worker 直连控制平面公网 IP（必须走 Overlay）
  - 入站拒绝: 所有（Worker 不接受入站连接）
```

### 8.4 审计日志

所有以下操作写入不可篡改的审计日志（Append-Only）：

- 用户登录/登出

- 管理员操作

## 十四、扩展性设计

### 14.1 横向扩展

- **控制平面**：无状态设计，可水平扩展。会话状态存 Redis，数据持久化存 PostgreSQL。
- **Worker 节点**：Worker 无共享状态，通过节点注册机制动态加入集群。新增节点自动承载新会话。
- **代理池**：独立服务，支持动态增减代理 IP，不影响运行中会话。

### 14.2 平台适配扩展

新增会议平台适配，只需：
1. 在 `apps/worker/src/agents/` 下新建 Agent 类，继承 `BaseAgent`
2. 在 `config/platforms.json` 中添加 URL 和 DOM 选择器配置
3. 在枚举类型 `PlatformType` 中增加新平台
4. 无需修改其他模块

### 14.3 多对多桥接扩展（N:M）

当前设计为 1:1 桥接（A 平台 ↔ B 平台）。未来可扩展为 N:M 模式：

```
腾讯会议（发起）
      │
      ▼
 中转混音节点 ──→ 钉钉（接入）
      │     ──→ 飞书（接入）
      │     ──→ 华为云（接入）
```

需要引入 GStreamer 或 Janus WebRTC 网关作为中央混流节点，而非当前的点对点管道。

## 十五、风险矩阵

| 风险 | 概率 | 影响 | 优先级 | 缓解措施 |
|:---|:---|:---|:---|:---|
| 平台 Web 端 DOM 结构变更导致脚本失效 | 高 | 中 | P1 | 选择器配置化；自动巡检脚本每日验证 |
| 平台封号（自动化检测） | 中 | 高 | P1 | 多账号池 + 住宅 IP + 人性化行为 |
| 住宅代理 IP 被会议平台列入黑名单 | 中 | 中 | P2 | 多供应商代理；IP 失效自动轮换 |
| v4l2loopback 内核模块在容器内不可用 | 低 | 高 | P2 | GStreamer appsrc 备用方案；宿主机预加载模块 |
| 音频回声导致会议体验差 | 中 | 中 | P2 | PulseAudio 单向隔离；软件 AEC（WebRTC AEC 库） |
| Worker 节点 CPU 过载（多路并发） | 中 | 中 | P2 | 并发上限 + 队列等待；节点自动扩容 |
| 平台 API/协议重大变更（如弃用 Web 端） | 低 | 极高 | P3 | 关注平台公告；备用方案：官方开放 SDK |
| 数据库单点故障 | 低 | 高 | P3 | PostgreSQL 主从 + 自动切换 |
| 凭证泄露 | 极低 | 极高 | P1 | AES-256-GCM 加密；密钥不落盘；审计日志 |
